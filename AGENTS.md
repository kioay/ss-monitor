# Project Rules

- After each implementation iteration, run the relevant local checks before handoff.
- Keep `.env`, credentials, cookies, deploy archives, `node_modules`, and build output out of Git.
- Production deploy archives must include a freshly generated `dist/` directory even though `dist/` stays out of Git. Use `scripts/create-deploy-archive.ps1` or an equivalent archive process; do not rely on the server to build frontend assets because production installs omit dev dependencies.
- Every completed iteration must be committed and pushed to the configured Git remote before reporting completion, unless the user explicitly asks not to push.
- Use concise commit messages that describe the shipped change.
- Do not rewrite shared history or force-push unless the user explicitly approves it.
- Do not send DingTalk test messages unless the user explicitly asks for a test in the current task. Test pushes are rate-limited by `DINGTALK_TEST_COOLDOWN_MINUTES`, but the rate limit is only a safety net and is not permission to test proactively.
- DingTalk robots send only the scheduled daily reports. Do not re-enable new-item, baseline, immediate, or 15-minute batched sentiment pushes unless the user explicitly asks to restore them.
- When changing production environment variables, persist them in `/opt/ss-monitor/.env` as the canonical baseline and mirror them into `/opt/ss-monitor/current/.env` before restarting. Release-local `.env` edits alone can be overwritten by the next deployment.
- Production now runs on the internal server `192.168.8.242`, which can reach Confluence directly. Persist the previous Confluence credentials in `/opt/ss-monitor/.env`, mirror them into `/opt/ss-monitor/current/.env`, and let the production service refresh `current-version-focus.json` itself. Do not rely on a daily local-workstation Confluence sync except as an explicit manual fallback.
- Use `scripts/deploy-ss-monitor-production.ps1` for one-command ss-monitor production deployments. It creates a fresh deploy archive through `scripts/create-deploy-archive.ps1`, uploads it to the internal production host, installs production dependencies, switches `/opt/ss-monitor/current`, mirrors `/opt/ss-monitor/.env` into the new release, restarts `ss-monitor`, and verifies `/api/health`.
- The one-command deploy script reads root SSH credentials only from `BETTAFISH_INNER_CREDENTIAL_FILE` or the indexed desktop credential file. Do not paste those credentials into Git, docs, release notes, issues, or chat.
- Production login credential index:
  - Keep this as an index only. Do not paste passwords, API keys, cookies, tokens, browser profiles, private keys, or generated credential payloads into Git, docs, issues, release notes, or chat.
  - Internal production host: `192.168.8.242`. Scripts that need root SSH can read `BETTAFISH_INNER_CREDENTIAL_FILE`; if unset, they look for `%USERPROFILE%\Desktop\内网机.txt` with a `root@<host>` line and a `password:<value>` line. Use the file only as a local credential source.
  - Canonical ss-monitor runtime credentials live only on the server in `/opt/ss-monitor/.env`, mirrored to `/opt/ss-monitor/current/.env` before restart. This is where production Confluence, Bilibili/Tieba cookies, DingTalk robots, Douyin remote-login password, and monitor runtime settings belong.
  - Production BettaFish runtime credentials live only on the server in `/opt/BettaFish/.env` and `/opt/BettaFish/current/.env`. Apply LLM/search credentials with `npm run apply:bettafish-credentials -- --dry-run` before `npm run apply:bettafish-credentials -- --restart`.
  - Local BettaFish credential staging uses the ignored `.env.bettafish-credentials.local` file, or an ignored file pointed to by `BETTAFISH_CREDENTIAL_ENV_FILE`. Relevant key groups are `BETTAFISH_SHARED_LLM_*`, the upstream engine triplets `REPORT_ENGINE_*`, `QUERY_ENGINE_*`, `INSIGHT_ENGINE_*`, `MEDIA_ENGINE_*`, plus `TAVILY_API_KEY` and either `ANSPIRE_API_KEY` or `BOCHA_WEB_SEARCH_API_KEY`.
  - SSH/deploy helper credentials must stay in ignored local environment files such as `.env.local`: `SYNC_CURRENT_VERSION_REMOTE`, `SYNC_CURRENT_VERSION_SSH_PORT`, and `SYNC_CURRENT_VERSION_PASSWORD` for public-target checks; `SYNC_BETTAFISH_*` / `SYNC_BETTAFISH_FULL_*` for BettaFish sync; `DOUYIN_SERVER_*` only for explicit server-side Douyin login tasks.
  - Douyin login state must be established on `192.168.8.242` through server-side MediaCrawler/CDP/noVNC/profile flows. The noVNC/VNC secret is indexed by `BETTAFISH_DOUYIN_REMOTE_PASSWORD` in the server `.env`; never copy workstation cookies or browser state to production.
- BettaFish on production server `192.168.8.242` must be deployed strictly according to upstream `https://github.com/666ghj/BettaFish` project requirements. Treat the upstream README, `.env.example`, `requirements.txt`, `docker-compose.yml`, `Dockerfile`, and MediaCrawler submodule as authoritative before changing deployment behavior.
- Production BettaFish on `192.168.8.242` must be a full upstream-compatible runtime under `/opt/BettaFish`: sync the complete repository and submodules, use Python 3.11/Python 3.9+ as upstream requires, install `requirements.txt`, install Playwright Chromium for crawler functionality, configure the root `.env` from upstream `.env.example`, and keep database/LLM settings aligned with upstream documentation.
- Do not replace the `192.168.8.242` BettaFish deployment with a lightweight subset or custom shortcut unless the user explicitly approves that exception. Keep `.env`, browser profiles, cookies, crawler output, downloaded media, generated reports, caches, and training datasets out of Git and deployment archives unless the user explicitly asks for a specific runtime artifact.
- For Douyin crawling on `192.168.8.242`, follow upstream BettaFish/MediaCrawler login behavior and persistence. Use server-side browser/CDP/profile or MediaCrawler-supported `qrcode`/`phone`/`cookie` flows as documented by the upstream code; do not copy local workstation cookies to production, and do not build unapproved workarounds that bypass Douyin's required manual verification.
- BettaFish semantic analysis is an auxiliary fusion signal, not a hard replacement for the SS1/SS2 domain analyzer. Do not let BettaFish sentiment alone create high risk; preserve protected contexts such as player skill sharing, player help requests, routine player sharing, and complaints about other players' behavior.
